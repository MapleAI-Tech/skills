---
name: chuxin
description: 产品大脑 — 项目初始化、每日简报、外部研究，驱动产品持续演进
---

<Purpose>
chuxin（初心）是产品的"第二大脑"。它通过理解项目现状、追踪迭代轨迹、研究外部世界，帮助开发者从"凭感觉提需求"进化到"有依据地规划演进"。它不是项目管理工具，而是产品思考伙伴。
</Purpose>

<Use_When>
- 用户在新项目中运行 `/chuxin init` — 初始化产品路线图
- 用户每天早上运行 `/chuxin` 或 `/chuxin briefing` — 获取昨日总结 + 今日规划引导
- 用户运行 `/chuxin research` — 触发外部竞品/趋势研究
- 用户说"帮我规划一下"、"今天该做什么"、"看看项目进度" — 自动匹配 briefing 模式
</Use_When>

<Do_Not_Use_When>
- 用户想直接写代码 — 用 ralph 或 autopilot
- 用户想修 bug — 直接修，不需要产品级规划
- 用户在做技术方案评审 — 用 plan 或 architecture
</Do_Not_Use_When>

<Project_Structure>
所有 chuxin 数据存储在项目根目录：

```
项目根目录/
├── ROADMAP.md              # 产品路线图（北极星，活的文档）
└── .chuxin/
    ├── journal/            # 每日研发日志
    │   └── YYYY-MM-DD.md
    └── insights/           # 外部研究洞察
        └── YYYY-MM-DD.md
```

**关键规则**：
- ROADMAP.md 和 .chuxin/ 始终在项目根目录，不在 docs/ 或其他子目录
- Skill 定义是全局的（用户级），数据是项目级的，天然隔离
- .chuxin/ 应加入 .gitignore（它是工作日志，不是项目资产）
- ROADMAP.md 应纳入版本控制（它是产品共识）
</Project_Structure>

<Commands>

### `/chuxin init` — 项目初始化

**核心原则：先理解再提问。绝不问能在代码中找到答案的问题。**

#### 步骤

1. **并行扫描** — 同时启动 4 个 Agent，每个 Agent 使用 `subagent_type="Explore"` 避免上下文爆炸：

   | Agent | 扫描内容 | 产出 |
   |-------|---------|------|
   | Agent A | 代码结构：技术栈、目录结构、关键文件、依赖 | 技术栈摘要 |
   | Agent B | 已有文档：CLAUDE.md、README、docs/、*.md | 文档知识摘要 |
   | Agent C | 功能现状：前端路由/页面、后端 API/数据模型、已完成的功能列表 | 功能清单 |
   | Agent D | Git 历史：近 30 天提交、开发节奏、活跃模块、进行中的工作 | 开发轨迹 |

2. **汇总分析** — 主 Agent 收到 4 份报告后，合成项目全貌：
   - 项目是什么（定位）
   - 技术栈和架构
   - 已完成的功能
   - 正在进行的工作
   - 开发节奏和模式

3. **引导式提问** — 基于分析结果，使用 `AskUserQuestion` 向用户提出**有针对性的选项**（不是开放式问题）：
   - "我看到项目定位是 X，对标产品可能是 A/B/C，你觉得更接近哪个？"
   - "目前已有 13 个页面，看起来处于 [阶段描述]，这个判断对吗？"
   - "最近的开发集中在 X 模块，接下来的重点你想继续深入还是转向 Y？"

   提问规则：
   - 每次只问一个问题
   - 提供具体选项而非开放填空
   - 先给判断再问确认（"看起来是 X，对吗？"）
   - 不问代码中能找到答案的问题

4. **生成 ROADMAP.md** — 基于用户的回答，生成路线图文件。格式：

   ```markdown
   # ROADMAP: {项目名称}

   ## 产品定位
   {一句话描述这个产品是什么、为谁解决什么问题}

   ## 对标产品
   - {产品 A}: {借鉴什么}
   - {产品 B}: {借鉴什么}

   ## 当前状态
   - 技术栈: {前端} + {后端}
   - 已完成: {功能列表}
   - 进行中: {当前工作}

   ## 演进路线

   ### 阶段一: {名称} ✅
   - [x] {已完成的里程碑}

   ### 阶段二: {名称} 🔄 (当前)
   - [ ] {里程碑 A}
   - [ ] {里程碑 B}
   - [ ] {里程碑 C}

   ### 阶段三: {名称}
   - [ ] {里程碑}

   ### 阶段四: {名称}
   - [ ] {里程碑}

   ## 竞品关注
   - {竞品名}: {关注点}

   ## 备注
   {其他重要信息，最后更新时间等}
   ```

5. **创建 .chuxin/ 目录结构**

6. **输出初始化报告** — 告诉用户：
   - ROADMAP.md 已生成，可以随时手动编辑调整方向
   - 下一步可以运行 `/chuxin` 开始每日简报

### `/chuxin` 或 `/chuxin briefing` — 每日简报

**前提：ROADMAP.md 存在。如果不存在，提示用户先运行 `/chuxin init`。**

#### 步骤

1. **信息收集** — 并行读取：
   - ROADMAP.md（定位 + 路线图）
   - 最近 3 天 .chuxin/journal/ 日志（迭代轨迹）
   - `git log --since="yesterday" --oneline`（昨晚的工作）
   - `git diff --stat HEAD~20`（近期改动范围）

2. **进度定位** — 对照 ROADMAP 的阶段和里程碑，标注当前进度：
   - 哪些里程碑已完成
   - 哪些在进行中
   - 当前处于路线图的什么位置

3. **输出今日简报**：

   ```markdown
   # 初心简报 — {日期}

   ## 昨夜工作摘要
   - {完成的事项，基于 git log}
   - {修改的模块/文件范围}

   ## 路线图进度
   当前位置: 阶段 {N} — {名称}
   - [x] {已完成的里程碑}
   - [ ] 🔄 {当前正在推进的里程碑}
   - [ ] {下一步里程碑}

   ## 发现的问题
   - {基于代码变更和日志发现的问题}

   ## 演进建议（按优先级）
   1. [P0] {建议} — 验收: {具体标准}
   2. [P1] {建议} — 验收: {具体标准}
   3. [P2] {建议} — 验收: {具体标准}

   ## 待你决定
   - {需要用户拍板的问题}
   ```

4. **引导对话** — 使用 `AskUserQuestion` 让用户：
   - 确认或调整优先级
   - 补充新想法
   - 决定今日重点

5. **生成今日日志** — 将对话结果写入 `.chuxin/journal/{今天日期}.md`：

   ```markdown
   # 研发日志 — {日期}

   ## 昨夜执行结果
   {摘要}

   ## 今日决策
   - {用户做出的决定}

   ## 今日计划
   1. [ ] {任务} — 验收: {标准}
   2. [ ] {任务} — 验收: {标准}

   ## 遗留问题
   - {未解决的问题}
   ```

6. **建议下一步** — 如果用户确认了计划，建议：
   - "运行 `/ralph` 或 `/plan` 开始今晚的开发规划"
   - "运行 `/chuxin research` 先研究一下 {竞品/技术}"

### `/chuxin research` — 外部研究

**前提：ROADMAP.md 存在，从中获取对标产品和竞品关注点。**

#### 步骤

1. **确定研究方向** — 基于 ROADMAP.md 中的：
   - 对标产品列表
   - 竞品关注点
   - 当前阶段的里程碑
   - 用户额外指定的方向（如果有参数）

2. **并行研究** — 使用多个 Agent 分别研究不同方向：
   - Agent 1: 竞品功能对比（Web 搜索）
   - Agent 2: 技术趋势（Web 搜索）
   - Agent 3: 用户痛点/社区讨论（Web 搜索）

3. **合成洞察报告** — 写入 `.chuxin/insights/{日期}.md`：

   ```markdown
   # 外部洞察 — {日期}

   ## 竞品动态
   - {竞品 A}: {新功能/变化}
   - {竞品 B}: {新功能/变化}

   ## 技术趋势
   - {趋势 A}: {对我们的影响}
   - {趋势 B}: {对我们的影响}

   ## 产品启发
   - {启发 1}: {可以借鉴什么}
   - {启发 2}: {可以借鉴什么}

   ## 建议纳入路线图
   - [ ] {建议}
   - [ ] {建议}
   ```

4. **引导用户决定** — 使用 `AskUserQuestion`：
   - 哪些建议要纳入路线图？
   - 哪些要加入今日计划？
   - 是否需要更新 ROADMAP.md？

</Commands>

<Tool_Usage>
- 使用 `Agent(subagent_type="Explore", ...)` 进行项目扫描，避免主上下文爆炸
- 使用 `AskUserQuestion` 进行交互式提问，提供具体选项
- 使用 `Read` 读取 ROADMAP.md、journal、insights 文件
- 使用 `Write` 创建 ROADMAP.md、journal、insights 文件
- 使用 `Bash` 运行 git log/diff 命令
- 使用 `WebSearch` 进行外部竞品和技术趋势研究
- 使用 `Glob` 和 `Grep` 扫描项目文件
- 多个 Agent 可以并行启动（init 的 4 个扫描 Agent、research 的研究 Agent）
</Tool_Usage>

<Examples>
<Good>
Init 时的引导式提问：
```
我看到项目是一个 AI 管理平台，已有 13 个页面，
技术栈是 React 19 + Go + SQLite。
最近的开发集中在 Chat 和实例管理模块。

从功能完整度看，基础 CRUD 已完成，
目前处于"功能深化"阶段。

这个判断准确吗？
```
Why good: 基于扫描结果给出判断，让用户确认而非从零开始描述。
</Good>

<Good>
Briefing 时的演进建议：
```
路线图显示你当前在"阶段二：运行时编排"，
昨晚完成了 reconciler 测试覆盖。

下一个里程碑是"实例生命周期完整闭环"，
建议今日重点：
1. [P0] 实例启动/停止状态机完善 — 验收: 所有状态转换有测试覆盖
2. [P1] 前端实例列表状态实时同步 — 验收: 状态变更后 3s 内 UI 更新
```
Why good: 有路线图锚点、有验收标准、有优先级。
</Good>

<Bad>
泛泛的开放式提问：
```
你的产品定位是什么？你想做什么功能？你的目标用户是谁？
```
Why bad: 这些都应该从代码和文档中推断，不该问用户。
</Bad>

<Bad>
没有验收标准的建议：
```
建议今天把文件管理做完。
```
Why bad: "做完"没有定义，无法收敛。必须有具体的验收条件。
</Bad>
</Examples>

<Escalation_And_Stop_Conditions>
- Init 时如果项目目录中没有代码（空项目），告知用户"chuxin 适用于已有代码基础的项目，请先初始化项目后再来"
- Briefing 时如果 ROADMAP.md 不存在，提示运行 `/chuxin init`
- Research 时如果无法访问网络，告知用户并基于已有信息生成报告
- 用户随时可以手动编辑 ROADMAP.md 调整方向，chuxin 每次运行都会重新读取
- 如果用户说"跳过"/"直接开始"，尊重用户意愿，简化流程
</Escalation_And_Stop_Conditions>

<Final_Checklist>
- [ ] 所有文件路径基于项目根目录（不是 docs/）
- [ ] 提问基于扫描结果而非凭空猜测
- [ ] 演进建议有验收标准（可验证）
- [ ] ROADMAP.md 每次运行时重新读取（不缓存旧版本）
- [ ] 日志和洞察按日期归档到 .chuxin/
- [ ] .chuxin/ 已建议用户加入 .gitignore
- [ ] 多 Agent 并行时使用 Explore 类型避免上下文爆炸
</Final_Checklist>
